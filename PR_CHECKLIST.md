# PR Checklist

다음 PR부터는 아래 항목을 기본 점검 기준으로 사용합니다.

## 1. 브랜치와 범위

- `main`으로 돌아가 최신 상태를 받은 뒤 새 브랜치에서 시작했는가
- 이번 브랜치에 이번 기능과 무관한 변경이 섞이지 않았는가
- PR 제목이 변경 범위를 한 문장으로 설명하는가
- `git diff main...HEAD` 기준으로 이번 기능만 들어있는가

## 2. 캐시

- 민감한 응답을 `fetch` 캐시, `revalidate`, `unstable_cache`에 올리지 않았는가
- 사용자별 데이터가 정적 생성이나 공유 캐시에 섞이지 않는가
- 토큰, 세션값, 개인정보가 URL 쿼리나 캐시 키에 들어가지 않는가
- 외부 CDN, 이미지, 에셋 URL에 내부 경로나 민감 파라미터를 붙이지 않았는가

## 3. 쿠키

- 쿠키에 토큰이나 민감 정보를 평문으로 저장하지 않았는가
- 인증 쿠키는 가능하면 `HttpOnly`, `Secure`, `SameSite` 정책을 갖추었는가
- 클라이언트 코드에서 `document.cookie`로 민감값을 읽거나 쓰지 않는가
- 로그, 에러 메시지, 디버그 출력에 쿠키 값이 찍히지 않는가

## 4. 토큰

- `NEXT_PUBLIC_*` 환경변수에 비밀값을 넣지 않았는가
- 액세스 토큰, 리프레시 토큰을 `localStorage`, `sessionStorage`에 저장하지 않았는가
- `Authorization` 헤더, Bearer 토큰, API 키가 클라이언트 코드에 하드코딩되지 않았는가
- 토큰이 URL, Referer, 에러 메시지, 분석 도구 로그에 남지 않는가

## 5. 에러 처리

- `console.error(err)` 등에 원본 에러 객체를 그대로 찍지 않았는가
- 사용자에게 보여주는 에러 메시지에 스택, 내부 경로, 토큰, 쿠키, 환경변수가 포함되지 않는가
- 서버 에러 응답에 내부 구현 정보 대신 일반화된 메시지를 사용했는가
- `catch(() => {})`로 에러를 무조건 삼키지 않고, 필요한 경우만 안전하게 마스킹해 처리했는가
- `error.tsx` 또는 공통 에러 처리 정책이 있는가

## 6. 외부 리소스

- 외부 이미지, 스크립트, 에셋 호스트가 정말 필요한 것만 허용되어 있는가
- 외부 요청에 민감 쿼리스트링이 붙지 않는가
- 제3자 서비스로 전달되는 메타데이터를 알고 있고 감수 가능한가

## 7. 최종 점검

- "이 브랜치에 이 기능만 있는지 확인해줘"라고 AI에게 마지막 검증을 맡겼는가
- 변경된 화면이나 기능에서 민감 정보가 브라우저 콘솔, 네트워크 탭, 에러 화면에 노출되지 않는지 확인했는가
